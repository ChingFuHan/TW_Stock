# Agent 執行任務單

## 任務名稱

台股券商 / 分行買賣超資料爬蟲與資料落地

## 任務目的

建立一套可重複執行的爬蟲，抓取特定券商 / 分行每日股票買進與賣出資料，清洗成結構化格式，作為後續跟單分析的原始資料來源。

## 核心原則

- 先求資料可穩定抓取，再求分析功能
- 先忠實保留原始資料，再做清洗與衍生欄位
- 不要假設網站參數與欄位意義，需以實際頁面驗證
- 發現結構不一致時，優先記錄與回報，不要靜默吞錯

## 已知來源

- `https://fubon-ebrokerdj.fbs.com.tw/z/zg/zgb/zgb0.djhtm?a=1030&b=1030`
- `https://fubon-ebrokerdj.fbs.com.tw/z/zg/zgb/zgb0.djhtm?a=8900&b=8900&c=E&d=1`

## 本次實作範圍

必做：

- 驗證範例網址可抓取
- 解析主要資料表
- 輸出原始 HTML
- 輸出清洗後表格資料
- 補上基本錯誤處理與日誌
- 提供可重複執行方式

不做：

- 自動下單
- 交易策略績效
- 視覺化 dashboard
- 即時盤中資料串接

## 建議專案結構

如專案尚未成形，優先整理成以下結構：

```text
.
├─ src/
│  ├─ fetcher.py
│  ├─ parser.py
│  ├─ cleaner.py
│  ├─ storage.py
│  └─ main.py
├─ data/
│  ├─ raw/
│  ├─ processed/
│  └─ logs/
├─ tests/
├─ README.md
└─ config/targets.example.json
```

若 repo 已有既有結構，優先沿用既有慣例，不要硬改目錄。

## 資料欄位目標

以頁面實際內容為準，但目標欄位至少整理為：

- `trade_date`
- `broker_code`
- `branch_code`
- `broker_name`
- `branch_name`
- `stock_code`
- `stock_name`
- `buy_lots`
- `sell_lots`
- `net_lots`
- `buy_amount`
- `sell_amount`
- `net_amount`
- `source_url`
- `fetched_at`

若頁面缺少其中欄位：

- 保留原始欄位
- 缺失欄位填 `null`
- 不可自行推造資料

## 執行階段

### Phase 1: 驗證來源

Checklist:

- [ ] 能成功下載 2 個範例網址
- [ ] 確認 HTTP 狀態碼、回應內容型態、頁面編碼
- [ ] 保存範例原始 HTML
- [ ] 確認資料是否真的存在於 HTML 內，而不是需要額外請求

輸出：

- 範例 HTML 檔
- 簡短驗證紀錄

### Phase 2: 解析頁面

Checklist:

- [ ] 找出主要資料表
- [ ] 確認表頭欄位名稱
- [ ] 處理中文編碼與數字格式
- [ ] 釐清每列代表的股票資料
- [ ] 釐清買進 / 賣出 / 淨額是否為獨立欄位或需計算

輸出：

- 可在單一頁面穩定運作的 parser
- 欄位對應說明

### Phase 3: 清洗與標準化

Checklist:

- [ ] 欄位命名統一
- [ ] 張數與金額欄位轉成數值型別
- [ ] 去除逗號、空白、特殊符號
- [ ] 明確定義 `net_lots`、`net_amount` 公式
- [ ] 保存原始欄位以便除錯

輸出：

- 清洗後 DataFrame 或等價資料結構
- 一份範例輸出檔

### Phase 4: 批次抓取

Checklist:

- [ ] 支援多個目標 URL 或券商清單
- [ ] 支援重試與逾時設定
- [ ] 失敗請求要寫入 log
- [ ] 成功請求要落地原始 HTML
- [ ] 避免同批資料重複寫入

輸出：

- 可批次執行的抓取流程
- 失敗與成功紀錄

### Phase 5: CLI 與文件

Checklist:

- [ ] 提供簡單命令列入口
- [ ] README 寫清楚安裝與執行方式
- [ ] 提供至少一個 targets 範例設定
- [ ] 說明輸出檔位置與格式
- [ ] 說明已知限制與風險

輸出：

- 可執行命令
- 使用說明

### Phase 6: 驗證

Checklist:

- [ ] 至少能完整跑過 2 個範例網址
- [ ] 輸出原始 HTML
- [ ] 輸出清洗後資料
- [ ] 對解析失敗有明確錯誤訊息
- [ ] 至少一個測試或驗證腳本

輸出：

- 驗證結果
- 測試或 smoke test 腳本

## Agent 實作要求

- 優先使用 Python
- 抓取、解析、清洗、保存分模組處理
- 不要把所有邏輯塞進單一腳本
- 若使用第三方套件，需補上依賴清單
- 若網站有反爬限制，先保守抓取，不要高頻請求

## 錯誤處理要求

至少處理以下情況：

- 連線失敗
- 回應為空
- 頁面編碼不符預期
- 找不到主要表格
- 欄位數量與預期不一致
- 數值轉換失敗

錯誤處理原則：

- 記錄來源 URL
- 記錄錯誤類型
- 記錄發生時間
- 保留足夠資訊方便重現

## 輸出規格

### 原始資料

建議命名：

- `data/raw/{date}/{broker_or_branch_id}.html`

### 清洗後資料

建議命名：

- `data/processed/{date}/broker_branch_flows.csv`

如資料量變大，可改用 Parquet。

### 日誌

建議命名：

- `data/logs/{date}.log`

## 驗收條件

只有在以下條件都成立時，才算完成：

- [ ] 範例網址可成功抓取
- [ ] 至少一個 parser 可穩定解析主要表格
- [ ] 有清洗後資料輸出
- [ ] 有原始 HTML 保存
- [ ] 有錯誤處理與 log
- [ ] 有可重複執行方式
- [ ] 有基本 README

## 若遇到不確定事項

處理順序如下：

1. 先看真實頁面內容
2. 再以程式輸出驗證欄位與數值
3. 最後才在文件中補上假設

不要做的事：

- 不要憑 URL 參數名稱自行解讀商業意義
- 不要補不存在的欄位值
- 不要在解析失敗時默默輸出空結果

## 最終交付物

至少要有：

- 可執行爬蟲程式
- 結構化輸出
- 原始 HTML 樣本
- README
- targets 範例
- 測試或驗證腳本

## 建議起手順序

1. 先用 2 個範例網址做最小可行抓取
2. 確認編碼與表格結構
3. 完成單頁解析
4. 再抽象成批次處理
5. 最後補 logging、README、驗證腳本
