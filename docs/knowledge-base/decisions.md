# 決策紀錄

本文件只記錄已經相對穩定、足以影響後續實作方向的決策。

## D-001 使用 Python 作為首選實作語言

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  爬蟲主體優先使用 Python。
- 理由：
  Python 在 HTTP 抓取、HTML 解析、表格清洗、資料輸出與後續分析上工具成熟，且符合目前需求文件方向。
- 影響：
  後續若建立專案骨架，優先以 Python 套件與 CLI 入口設計。

## D-002 先保留原始 HTML，再做結構化清洗

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  每次抓取需保留 raw HTML，不能只保留清洗後結果。
- 理由：
  目前未知頁面結構、編碼與欄位穩定性。保留原始內容能支援除錯、重跑 parser、驗證欄位來源。
- 影響：
  後續 storage 設計需同時支援 raw 與 processed 輸出。

## D-003 先驗證真實頁面，再解讀參數與欄位

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  不依據 URL 參數名稱直接假設商業意義，所有欄位與參數含義需從頁面結果反推或驗證。
- 理由：
  目前只有範例網址，沒有正式規格文件。先假設容易導致 parser 與 schema 走偏。
- 影響：
  實作初期應聚焦在「抓下來、看真實結構、再建模」。

## D-004 程式需拆成抓取、解析、清洗、保存模組

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  不把全部邏輯寫在單一腳本中。
- 理由：
  後續一定會有欄位修正、來源擴充、批次處理與測試需求。模組化可以降低修改成本。
- 影響：
  建議至少拆成 fetcher / parser / cleaner / storage / entrypoint。

## D-005 建立 repo 內建知識傳遞庫

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  在 `docs/knowledge-base/` 下建立可持續更新的知識庫，而不是只保留單次摘要。
- 理由：
  這個專案會有多輪 agent 接手。需要固定入口、日誌、決策與交接機制，避免每次從零理解。
- 影響：
  後續每次實作都應同步回寫知識庫。

## D-006 第一版優先使用 Python 標準庫完成

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  第一版抓取、解析、輸出與測試優先只用 Python 標準庫完成。
- 理由：
  目前環境沒有 `requests`、`pandas`、`bs4`、`lxml` 等套件。先用標準庫完成最小可行版本，可以降低安裝與部署阻力，並先驗證真實頁面結構。
- 影響：
  目前實作以 `urllib.request`、`html.parser`、`csv` 為主。未來若要改用 `pandas` 等工具，需確認其收益是否高於新增依賴成本。

## D-007 以真實 table 結構而非表頭文字決定資料意義

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  不能因表頭第一欄顯示「券商名稱」就把資料列當成券商。實際資料列以股票為主，需根據真實列內容解析。
- 理由：
  真實頁面中資料列使用 `GenLink2stk(...)` 或股票 anchor，明確指向股票代碼與股票名稱。
- 影響：
  parser 需要從資料列本身抽取 `stock_code` / `stock_name`，不能直接依賴欄位名。

## D-008 券商 / 分點名稱由 lookup JS 解析

- 日期：2026-04-07
- 狀態：Accepted
- 決策：
  `broker_name` 與 `branch_name` 以 `/z/js/zbrokerjs.djjs` 為主要名稱來源。
- 理由：
  原始 HTML 只提供 `GenFundCorpCombo('broker','branch','frm')` 中的 code，不直接帶出可穩定解析的名稱。
- 影響：
  若 lookup JS 抓取失敗，主流程仍可退回 code-only，但名稱欄位會留空。
