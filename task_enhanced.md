# 台股券商分點買賣資料爬蟲任務書

## 1. 任務目標

建立一套可執行的爬蟲流程，抓取台股「券商 / 分行」每日股票買入與賣出資料，並整理成可分析的結構化資料，作為後續「跟單指標」的基礎資料來源。

此任務的核心不是交易，而是先把資料穩定抓下來、清洗完成、可重複更新、可供分析。

## 2. 背景

我想追蹤特定券商或分點每天對各檔股票的買入、賣出與淨流向，觀察是否能作為市場上有參考價值的籌碼訊號。

因此第一階段要先完成：

- 穩定抓取資料
- 正確解析表格
- 保存為可查詢格式
- 後續可擴充成指標計算或策略分析

## 3. 資料來源

目前已知來源如下：

- 範例 1: `https://fubon-ebrokerdj.fbs.com.tw/z/zg/zgb/zgb0.djhtm?a=1030&b=1030`
- 範例 2: `https://fubon-ebrokerdj.fbs.com.tw/z/zg/zgb/zgb0.djhtm?a=8900&b=8900&c=E&d=1`

## 4. 任務範圍

本階段只處理「資料抓取與整理」，不包含自動下單。

需要完成的事情：

- 能下載指定券商 / 分行頁面的內容
- 能解析頁面中的股票買進 / 賣出資料
- 能把結果轉成結構化資料
- 能保存原始資料與清洗後資料
- 能支援後續批次擴充

暫不要求：

- 即時報價整合
- 自動交易
- 視覺化 dashboard
- 回測系統

## 5. 輸入與輸出

### 輸入

至少支援以下輸入：

- 券商代碼或分行代碼
- 單一查詢 URL
- 多個查詢目標清單

若網站實際支援日期、市場別或其他查詢參數，也請保留可擴充空間，但不要先假設所有參數意義，需以頁面實際內容驗證。

### 輸出

至少產出以下內容：

- 原始 HTML
- 結構化表格資料
- 執行日誌
- 簡單的抓取摘要

建議輸出格式：

- `data/raw/`：原始 HTML
- `data/processed/`：清洗後 CSV 或 Parquet
- `data/logs/`：抓取 log

## 6. 預期欄位

請以頁面實際可取得欄位為準，但至少要盡量整理成以下結構；若某欄在頁面不存在，可留空或標記 `null`：

- `trade_date`：資料日期
- `broker_code`：券商代碼
- `branch_code`：分行代碼
- `broker_name`：券商名稱
- `branch_name`：分行名稱
- `stock_code`：股票代碼
- `stock_name`：股票名稱
- `buy_lots`：買進張數
- `sell_lots`：賣出張數
- `net_lots`：淨買賣張數
- `buy_amount`：買進金額
- `sell_amount`：賣出金額
- `net_amount`：淨買賣金額
- `source_url`：來源網址
- `fetched_at`：抓取時間

如果頁面只提供部分欄位，請忠實保留原始欄位，避免為了對齊格式而捏造數據。

## 7. 功能需求

### 7.1 抓取

- 能針對單一 URL 抓取內容
- 能針對多個券商 / 分行批次抓取
- 具備基本重試機制
- 具備逾時處理
- 具備失敗記錄

### 7.2 解析

- 能辨識頁面主要資料表
- 能正確處理中文編碼問題
- 能處理數字格式，例如逗號、正負值、空白、缺值
- 若頁面結構有變動，至少要能回報錯誤位置，而不是靜默失敗

### 7.3 清洗

- 統一欄位命名
- 數值欄位轉成可計算型別
- 保留原始文字欄位以利除錯
- 產生 `net_lots` 與 `net_amount` 時，必須明確標示公式與資料來源欄位

### 7.4 保存

- 每次抓取保留原始 HTML
- 清洗後資料可累積保存
- 避免重複寫入同一筆資料

### 7.5 可維護性

- 程式結構應拆分為抓取、解析、清洗、保存等模組
- 重要邏輯需要有最基本的測試或驗證腳本
- 需要提供簡單執行方式，例如命令列入口

## 8. 非功能需求

- 優先使用 Python 實作
- 程式需能在本機重複執行
- 命名清楚，避免把商業規則寫死在單一腳本
- 需要保留擴充空間，未來可接更多券商來源

## 9. 建議實作方式

可接受的方向：

- 使用 `requests` 或其他穩定 HTTP client 抓取頁面
- 使用 `BeautifulSoup`、`pandas.read_html` 或其他合適方式解析表格
- 使用 `pandas` 進行清洗與輸出

若網站有特殊編碼、反爬限制、或資料需透過特定參數才能取回，請在實作中清楚記錄假設與限制。

## 10. 例外與風險

實作時請特別注意：

- 網頁可能使用 Big5、UTF-8 或混合編碼，需驗證後處理
- 同一頁面可能有多個表格，不能抓錯表
- 參數 `a`、`b`、`c`、`d` 的意義不能憑空假設，需以頁面結果驗證
- 有些欄位可能不是固定存在
- 網站可能有限流或阻擋，需保守抓取頻率

## 11. 驗收標準

完成後至少要能達成：

- 成功抓取上述 2 個範例網址
- 能把頁面中的主要股票買賣資料解析成結構化表格
- 能輸出至少一份清洗後資料檔
- 能保存對應原始 HTML
- 對抓不到、解析失敗、欄位缺失等情況有明確錯誤訊息
- 有基本使用說明

## 12. 建議交付物

交付內容至少包含：

- 可執行爬蟲程式
- 資料解析模組
- 輸出資料目錄
- `README.md`
- 範例執行方式
- 至少一份測試或驗證腳本

## 13. Agent 執行指示

請依照以下順序進行：

1. 先驗證範例網址是否可成功取得內容。
2. 確認頁面編碼、表格結構、欄位名稱。
3. 先完成單一頁面的穩定解析。
4. 再擴充成可批次抓取。
5. 設計資料輸出格式，並保留原始 HTML。
6. 補上錯誤處理、重試、日誌與基本說明文件。

## 14. 完成定義

只有在以下條件成立時，才算任務完成：

- 能重複執行
- 能穩定抓到資料
- 能輸出結構化結果
- 能追查錯誤原因
- 後續能直接拿來做分點跟單分析

## 15. 補充說明

如果頁面中的某些欄位無法直接取得，優先策略是：

- 先完整保留原始資料
- 再標記缺失欄位
- 不要為了湊欄位而假造值

如果網站實際結構與目前推定不一致，請以「真實頁面結構」為主，並同步更新文件與欄位定義。
